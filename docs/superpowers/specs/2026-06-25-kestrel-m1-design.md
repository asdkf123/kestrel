# Kestrel — 자작 브라우저 엔진: M1 설계 문서

- 날짜: 2026-06-25
- 상태: 승인됨 (브레인스토밍 완료)
- 범위: 마일스톤 1 (M1). 전체 로드맵은 맥락용으로 함께 기록.

## 1. 프로젝트 개요

크로미움/WebKit/Gecko 같은 기존 엔진을 쓰지 않고 **밑바닥부터 직접 만드는 브라우저 엔진**. 코드네임 **Kestrel**(황조롱이 — 작고 빠른 매. "가볍고 빠르고 메모리 적게"라는 컨셉을 상징).

- 언어: **Rust** (런타임 GC 없음 → 가볍고 빠르고 저메모리. 메모리 안전성은 컴파일러가 강제 → 다년 1인 프로젝트의 생존율을 높임).
- 성격: **장기 자작 엔진 프로젝트**. Ladybird/Servo처럼 처음부터 끝까지 직접.
- 개발자 배경: 현재 주력 언어는 TypeScript/Next.js. Rust는 이 프로젝트로 **처음부터 배우며** 진행한다. 따라서 마일스톤은 "Rust 학습 + 눈에 보이는 결과"가 함께 나오도록 작게 끊는다.

### 솔직한 기대치 (중요)

- 크롬/파이어폭스/사파리는 수백~수천 명이 십수 년간 만든 엔진이다. "그 수준"을 1인이 따라잡는 건 비현실적이며 목표가 아니다.
- 현실적이고 도달 가능한 목표는 **"내가 직접 만든 엔진으로 실제(에 가까운) 웹페이지를 화면에 그린다"**. Ladybird가 1인 취미 프로젝트로 시작해 독립 엔진이 된 선례가 있다.
- "가볍다"는 기능을 적게 지원하는 대신 충분히 달성 가능하다. 다만 **자바스크립트 실행 속도**만큼은 크롬이 그 무게로 사는 부분이라 거기서 크롬을 이기는 건 목표가 아니다.

## 2. 전체 로드맵 (맥락)

각 마일스톤은 "완성하면 눈에 보이는 결과"가 나오도록 끊는다. **이 문서는 M1만 상세 설계한다.**

- **M0 — 토대**: Rust 툴체인 셋업, 기초 문법 익히기. (M1과 통합 진행)
- **M1 — 렌더링 파이프라인 미니어처 (이 문서)**: 작은 부분집합의 HTML+CSS → 창에 색깔 박스.
- **M2 — 텍스트 + 더 풍부한 레이아웃**: 폰트 래스터화, 인라인 레이아웃, 줄바꿈. "웹페이지처럼" 보이는 첫 순간.
- **M3 — 네트워킹 + 더 많은 CSS**: HTTP(S)로 실제 URL 가져오기, 셀렉터/속성 확장, flexbox 일부, 이미지.
- **M4 — 자바스크립트**: 처음엔 Rust 네이티브 JS 엔진(`Boa`)을 임베드. DOM API 노출, 이벤트. (JS 엔진 자작은 후순위 — 별개의 다년 과제)
- **M5+ — 실제 사이트 렌더링을 향해 점진 확장.**

## 3. M1 목표 (한 줄)

작은 부분집합의 HTML과 CSS 파일을 입력하면, 그 구조대로 **중첩된 색깔 박스**가 OS 창에 실시간으로 그려진다. (글자 렌더링은 M2)

## 4. 아키텍처

브라우저의 핵심 흐름을 작게 한 바퀴 도는 파이프라인. 각 모듈은 입력 타입 하나를 받아 출력 타입 하나를 내놓고, 독립적으로 테스트된다.

```
바이트 → [html] → DOM 트리 → [style] → 스타일 트리 → [layout] → 레이아웃 트리 → [paint] → 픽셀 버퍼 → [window] → 화면
                     ↑
        [css] → 스타일시트 ─┘
```

### 모듈 분해

| 모듈 | 입력 → 출력 | 책임 |
|------|------------|------|
| `dom` | — | `Node` 타입 정의 (Element / Text) |
| `html` | `String` → `Node` 트리 | HTML 토크나이저 + 파서 |
| `css` | `String` → `Stylesheet` | CSS 토크나이저 + 파서 (Rule = 셀렉터들 + 선언들) |
| `style` | DOM + Stylesheet → `StyledNode` 트리 | 셀렉터 매칭, 각 노드에 계산된 속성 맵 부착 |
| `layout` | StyledNode 트리 → `LayoutBox` 트리 | 블록 레이아웃 + 박스 모델 (각 박스의 x/y/width/height 계산) |
| `paint` | LayoutBox 트리 → `Vec<u32>` 픽셀 버퍼 | 배경색 사각형들을 버퍼에 칠하기 |
| `window` | 픽셀 버퍼 → 화면 | winit + softbuffer로 창에 표시 |

설계 원칙: 각 모듈은 "무엇을 하는가 / 어떻게 쓰는가 / 무엇에 의존하는가"가 명확해야 하고, 내부 구현을 바꿔도 소비자가 깨지지 않아야 한다. 데이터 타입(`Node`, `Stylesheet`, `StyledNode`, `LayoutBox`)이 모듈 간 계약이다.

### 데이터 흐름

`bytes → DOM → StyledNode 트리 → LayoutBox 트리 → 픽셀 버퍼 → 창`

## 5. M1 범위 (의도적으로 아주 작게)

"전체 흐름을 한 번 완성"이 목표라서 각 단계는 최소 부분집합만 지원한다.

- **HTML**: 여는/닫는 태그, 속성(`class`, `id`), 텍스트 노드, 중첩. (주석/엔티티/self-closing 태그는 제외 또는 최소)
- **CSS**: 태그/클래스/id 셀렉터. 속성은 `display`, `width`, `height`, `margin`, `padding`, `background-color`. 단순 specificity 기반 캐스케이드.
- **Layout**: **블록 레이아웃만** (세로로 쌓이는 박스 + 박스 모델). 인라인/텍스트 배치는 M2.
- **Paint**: 사각형 채우기만. 글리프 없음.

→ M1 출력 = "중첩된 색깔 박스들". 텍스트는 M2에서 도입.

### 명시적 비범위 (M1에서 하지 않음)

- 텍스트/폰트 렌더링, 인라인 레이아웃 (M2)
- 네트워킹/URL (M3) — M1은 로컬 파일만 읽음
- 자바스크립트 (M4)
- flexbox/grid, position, float, z-index, 이미지 (후속)

## 6. 의존성 원칙

학습이 목적이므로 **파서/스타일/레이아웃/페인트는 전부 직접 구현**한다(여기가 배우는 핵심). 창 띄우기만 직접 구현이 비현실적이라 외부 크레이트 사용:

- `winit` — OS 창 + 이벤트 루프
- `softbuffer` — CPU 픽셀 버퍼를 창에 복사 (GPU/셰이더 불필요)

## 7. 에러 처리

- **파서는 관용적으로**: 깨진 입력에도 패닉 대신 부분 트리를 반환한다(실제 브라우저 철학). M1은 학습용이라 복구 로직은 단순하게.
- 예상 못한 내부 불변식 위반은 패닉 허용(개발 중 빠른 발견).

## 8. 테스트 전략

- 각 모듈 단위 테스트. Rust의 `#[test]` 사용.
  - 파서: "입력 문자열 → 기대 트리" 비교.
  - 레이아웃: "입력 트리 → 기대 박스 좌표" 비교.
- TDD 적용 가능(Rust 학습에도 좋은 소재).

## 9. 프로젝트 구조

- 위치: library-manager와 **별개의 새 프로젝트** → `~/Documents/Projects/kestrel/`

```
kestrel/
  Cargo.toml
  src/
    main.rs       # 파일 읽기 → 파이프라인 → 창 띄우기

    css.rs

    layout.rs
    paint.rs
    window.rs
  examples/
    test.html
    test.css
```

## 10. M1 완료 기준 (Definition of Done)

1. `examples/test.html` + `examples/test.css`(중첩 div + 배경색/크기/여백)를 입력으로 실행하면,
2. OS 창이 뜨고,
3. HTML 구조와 CSS에 맞게 **중첩된 색깔 박스**가 올바른 위치/크기로 그려진다.
4. html/css/style/layout 모듈에 단위 테스트가 존재하고 통과한다.
