# Kestrel 성능 기준 (Performance Budget)

> Kestrel의 정체성은 **작고, 빠르고, 무겁지 않고, 좋은** 브라우저다.
> 이 문서는 그 정체성을 **측정 가능한 목표치**로 못박는다. 모든 마일스톤은 이 잣대에 비춰진다.
>
> 주의: 아래 숫자는 대부분 아직 *목표(target)*이지 측정값이 아니다. 진짜 페이지와 벤치마크 하니스가 생기면 보정한다. 핵심은 "의지가 아니라 데이터로 가벼움을 지킨다"이다.

## 1. 메모리 (가벼움 / 무겁지 않음)

| 항목 | 목표 | 스트레치 | 비교 |
|------|------|----------|------|
| 앱 + 단순 페이지 1개 (RSS) | ≤ 50MB | ≤ 30MB | 크롬 브라우저+탭1 흔히 300~600MB. 경량(Dillo/NetSurf) 수십 MB |
| 탭 추가당 | ≤ 20MB | — | 크롬 탭=프로세스, 탭당 50~150MB+ |
| 백그라운드 탭 | ≤ 수 MB (렌더 트리 버리고 DOM+스크롤 상태만) | — | — |
| 메모리 해제 | 탭 닫으면 즉시 회수, 캐시 상한 고정, 누수 0 | — | Rust(GC 없음) 결정적 해제 |

## 2. 속도 (빠름) — RAIL 모델 기반

| 항목 | 목표 | 근거 |
|------|------|------|
| 콜드 시작 → 창 표시 | ≤ 300ms | 크롬 콜드 스타트 ~1s+ |
| 정적 페이지 첫 렌더 (파싱→페인트, ~50KB) | ≤ 50ms | — |
| 스크롤 / 애니메이션 | 60fps (≤ 16ms/프레임) | 증분 레이아웃·페인트 전제 |
| 입력 → 화면 반응 | ≤ 100ms | 구글 RAIL 100ms 규칙 |

## 3. 크기 (작음)

| 항목 | 목표 | 비교 |
|------|------|------|
| 릴리즈 바이너리 | ≤ 25MB | 크롬 설치본 수백 MB |
| 직접 의존성 | 최소 (현재 winit + softbuffer, 엔진 로직 0-dep) | V8만 수십 MB |

## 4. 품질 / 호환 (좋음)

- **골든 페이지 셋**: 정해둔 페이지 묶음이 픽셀 단위로 올바르게 렌더 (시각 회귀 테스트). 작게 시작해 키운다.
- **타깃 사이트 부류**: 정적/콘텐츠 사이트 (블로그, 문서, 위키, 뉴스 기사). 무거운 SPA(피그마/구글시트)는 **초기 타깃 아님**. 이 선을 명시하는 것이 가벼움의 핵심.
- **안정성**: 골든 셋에서 크래시 0.
- **프라이버시**: 텔레메트리/백그라운드 통신 없음. 경량 브라우저가 이기는 지점.

## 5. 메타 기준 — 어떻게 지키나

- 마일스톤마다 고정 페이지 셋에서 **메모리 + 렌더 시간 기록**. 회귀 추적.
- 기능마다 비용 측정. 정당한 이유 없이 메모리/시간을 임계 이상 악화시키는 변경은 거부.
- 측정 하니스가 갖춰지면(네트워킹으로 실제 페이지 로딩 가능해지는 M3 무렵) CI에서 예산 강제.
